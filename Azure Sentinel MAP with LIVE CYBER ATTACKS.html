Azure Sentinel MAP with LIVE CYBER ATTACKS

The Results

I successfully logged all the failed RDP attempts to log into my azure virtual machine.

Although there were many attempts from several countries, the IP address from the Netherlands made a significant attempt to brute force into the machine.

All login attempts were logged and entered into a powershell script to look up the attackers Geolocation information and plot it on the Azure Sentinel Map


All the steps I took to achieve this result

Our first step is to create a virtual machine in Azure and then in the networking I created an inbound rule that would allow connections from any port range (*). This will allow all traffic into our virtual machine making it a perfect honeypot for our attackers. Also since every port is available hopefully attackers will find the machine quickly.



This was the final settings I used for the virtual machine.


Our next step is to create a log analytics workspace. The purpose of this is to create a log that can track the geographic information of connection attempts to our machine.


You can see the settings I chose below.


Next we need to enable logging through the machine so we can actually track the information through the log analytics workspace. To do this I made sure to enable the Foundational CSPM and servers. However, SQL servers are not needed for this task so I disabled that.


I then made sure to make sure that we were storing all windows events but turning on the setting below.



Once the logging was set up correctly I then connected my virtual machine to the log analytics workspace.



Once the virtual machine was set up correctly to log events I made sure to add azure sentinel to the virtual machine. This way when the log events come through the machine we have a tool to help us organize and sort through the information.


Once I attached Sentinel to our VM I wanted to make sure I could log into the machine via RDP. After inputting the correct login information I was able to get in!


After logging in I wanted to make sure I was understanding and capturing the right event for incorrect logins to the machine. In order to do this I purposely attempted to RDP into the machine with the incorrect information so I could grab the login event. When looking inside the event viewer I was able to determine that the event was 4625. An important piece of information in this event log is the IP address, we can use this to determine the geolocation of the login attempt.


Using https://ipgeolocation.io/ we are able to determine where the IP address is coming from.


But before I am able to get any attackers to find my machine I have to allow ICMP echo requests by turning off my firewall. Below you can see pings to my virtual machine before and after turning off my firewall. Once the firewall has been turned off it can receive ICMP echo requests.




Many times with machines like this the attacker will try to brute force into the machine. While trying hundreds or thousands of login attempts it will become very tedious to check the geolocation of each attempt manually.


Because of this we need to use a powershell script that allows us to automatically log the geolocation of the login attempt via API.


Once I input my API key from ipgeolocation.com I was able to get the script up and running. Below you can see the login attempt tests from my machine.

The script will also note this information and export it into a log file on the virtual machine which we will be able to pick up in azure and analyze with sentinel.

In order to get sentinel the data it needs for analysis we need to create a custom log. When creating the custom log in azure I uploaded the log that the powershell script created in the sample log section in order to train azure. Additionally inside the virtual machine we selected the collection path to be where the powershell script created the log. This way the log updates from additional login attempts and we can see it in sentinel.
